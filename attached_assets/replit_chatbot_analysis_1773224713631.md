# รายงานวิเคราะห์การทำงานและปัญหาของ Chatbot (สำหรับ Replit/ทีมพัฒนา) - สรุปหลังจากได้รับ Feedback รอบ 2

จากการได้รับ Feedback เพิ่มเติมจากทีมงาน Replit และการตรวจสอบการทำงานของ `_bot_chat_reply()` เชิงลึก นี่คือข้อสรุปปัญหาและแนวทางการแก้ไขที่สมบูรณ์ที่สุดที่นำไปใช้ในการปรับปรุงโค้ดครับ

## 🛑 จุดอ่อนและปัญหาที่พบในโค้ด

ข้อผิดพลาดที่ทำให้บอทสูญเสียบริบทการสนทนาเกิดจาก **4 ปัจจัย** ที่ทำงานร่วมกัน:

### 1. การล้างบริบท (Context Reset) ที่รุนแรงไปเพราะขาดการคัดกรองคำทั่วไป (Stop Words)
*   **ปัญหาที่เกิดขึ้น:** ลอจิกการเคลียร์บริบทด้วย N-gram นั้นกว้างเกินไป เพราะนำทุกคำที่ลูกค้าพิมพ์ไปหั่นเป็นชิ้นๆ แล้วค้นหา คำทั่วไปในบริบทของร้าน (เช่น "พยาบาล", "EKG", "เสร็จแล้ว") เมื่อนำไปค้นหาจะไปตรงกับชื่อของสินค้าอื่นแทบทุกชิ้นในร้าน ทำให้ไปเข้าเงื่อนไข `hits_current == 0` (เพราะเป็น generic hit แบบ count > 3) แต่กลับไปทะลุเงื่อนไขถัดไปที่ไปเจอสินค้าอื่น ทำให้ระบบสั่งล้างบริบทเดิมทิ้งทันที
*   **การแก้ไขที่นำไปใช้:** 
    * ทำการกรองคำโดเมนเฉพาะทางออกก่อนสร้าง N-gram: ตัดคำว่า `'พยาบาล', 'ekg', 'ekgshops'` ออกจากข้อความลูกค้า 
    * ตั้งค่าความยาว N-gram ให้ Trigger ได้เมื่อมีความยาวขั้นต่ำ 4 ตัวอักษรขึ้นไป

### 2. เมื่อมี Context (current_product_id) ยังปล่อยให้ Keyword Search ทำงานกวนบริบท
*   **ปัญหาที่เกิดขึ้น:** นี่เป็นเรื่องสำคัญหากเจตนาคือการเกาะ context เดิมไปเรื่อยๆ โดยที่ผ่านมาถ้าผู้ใช้พิมพ์ข้อความแล้ว Keyword Search ข้างล่างดันทำงาน มันจะดึงสินค้านั้นมารวมใน `products_text` ดึงความสนใจ AI ออกจากสินค้าหลัก (จุดนี้มีการเข้าใจผิดเล็กน้อยในตอนแรกว่า Guest bot ปิด keyword search ไปแล้ว แต่แท้จริงนั่นเป็นเพียงส่วน Fallback)
*   **การแก้ไขที่นำไปใช้:** เมื่อ `current_product_id` ยังมีค่าอยู่ (คือลูกค้าไม่ได้ตั้งใจเปลี่ยนเรื่อง และไม่ถูก N-gram ตัดทิ้ง) ระบบจะ Query เฉพาะสินค้านั้นพร้อมรายละเอียดเจาะลึก (`detailed=True`) ลงตัวแปร `prods` เป็น Array และข้ามการค้นหาด้วย Keyword จากคำพูดใดๆ ในตาข่ายนั้นไปเลย หากต้องการให้ลูกค้าเปลี่ยนบริบท ต้องรอให้ N-gram ขั้นตอนบนสุดเป็นผู้ตัดสินและส่งต่อมาให้บล็อคค้นหานี้ทำงาน

### 3. AI State Overwrite ส่ง `current_product_id` เป็น null ทับซ้อนระบบ (จุดบอดที่เพิ่งพบ)
*   **ปัญหาที่เกิดขึ้น:** มีจุดที่ทั้ง 2 ฝ่ายเพิ่งมองเห็น คือ `session_data` ถูกอัปเดตโดย AI ผ่าน JSON return ในตัวแปร `new_state` โค้ด Backend มีการทำ Merge ตัวแปรเข้าหากัน ถ้า Gemini ตอบกลับมาพร้อม `{"new_state": {"current_product_id": null}}` ด้วยการตัดสินใจเองว่าจบบทสนทนาเรื่องชุดแล้ว มันจะไปทับผลของการปกป้องจาก N-gram guard ทันที ทำให้บริบทถูกลบออกไปดื้อๆ
*   **การแก้ไขที่นำไปใช้:** เพิ่มกฎเหล็กอุดช่องโหว่นี้ใน System Prompt คือ ห้าม AI ลบ (`set เป็น null`) ตัวแปร `current_product_id` ใน `new_state` เด็ดขาด เว้นแต่ลูกค้าจะสั่งความต้องการเปลี่ยนสินค้าหรือหมวดหมู่อย่างโจ่งแจ้งแล้วเท่านั้น

### 4. Prompt Rule เดิมบังคับ "ห้ามใช้ประวัติแชท" อย่างขวานผ่าซาก
*   **ปัญหาที่เกิดขึ้น:** กฎเดิมระบุว่าให้ `❌ ห้ามใช้ประวัติแชทเดิมเป็นข้อมูลสินค้า ต้องอ้างอิงจากรายการสินค้าด้านล่างเท่านั้น` ซึ่งความตั้งใจคือลดการมโนข้อมูลราคาและสต็อค แต่จังหวะที่ Backend ล้างบริบทพลาดและส่งรายชื่อสินค้าแบบสุ่มมาให้ กฎนี้กลับสั่งให้ AI ทิ้งประวัติแชทซะ ทำให้มันลืมความต้องการแรกเริ่มทันที
*   **การแก้ไขที่นำไปใช้:** เปลี่ยนเป็นคำสั่งเชิงบวกที่แยกแยะการใช้งานชัดเจน: 
  > `"✅ ให้ใช้ประวัติแชทเพื่อทำความเข้าใจบริบทว่าลูกค้ากำลังถามถึงสินค้าใด แต่ข้อมูล สต็อก/ราคา/รายละเอียดสินค้า ต้องดึงมาจากรายการสินค้าด้านล่างเท่านั้น ห้ามสร้างข้อมูลขึ้นเอง"`

---

## 🛠️ สรุป

เป้าหมายสูงสุดเพื่อจัดการกับทั้ง 4 ข้อถูก Apply ลงใน `app.py` แล้ว โค้ด N-gram และเงื่อนไขการเลือกแสดงสินค้า ล้วนแต่สอดประสานกันเพื่อล็อกให้ Gemini ทราบความต้องการปัจจุบัน และลดข้อผิดพลาดในการเผลอลืมสถานะเก่าให้เหลือน้อยที่สุด พร้อมส่งโค้ดนี้ขึ้น production ได้เลยครับ
