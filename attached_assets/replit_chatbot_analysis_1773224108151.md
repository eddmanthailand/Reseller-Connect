# รายงานวิเคราะห์การทำงานและปัญหาของ Chatbot (สำหรับ Replit/ทีมพัฒนา) - ฉบับปรับปรุง

จากการตรวจสอบโค้ดในส่วน `app.py` (ฟังก์ชัน `_bot_chat_reply`) และนำข้อเสนอแนะจากทีมงาน Replit/ผู้พัฒนามาประกอบการวิเคราะห์ นี่คือข้อสรุปสาเหตุที่บอทสูญเสียบริบทการสนทนาและคำแนะนำในการแก้ไขครับ

## 🛑 จุดอ่อนและปัญหาที่พบในโค้ด (เรียงตามความสำคัญ)

### 1. การล้างบริบท (Context Reset) ที่รุนแรงและกว้างเกินไปจากการใช้ N-gram Search
(อ้างอิง `app.py` ประมาณบรรทัดที่ 16141-16179)
โค้ดมีลอจิกในการตรวจจับการเปลี่ยนเรื่อง (Context-switch detection) โดยนำข้อความล่าสุดมาหั่นเป็น N-gram แล้วเช็คว่าตรงกับสินค้าปัจจุบันหรือไม่ และถ้านำไปหาใน Database เจอตรงกับสินค้าอื่น จะทำการล้างบริบททันที (`current_product_id = None`)

*   **สิ่งที่เกิดขึ้นจริง (ตามความเห็นทีม Replit):** ลอจิกการเคลียร์บริบทนี้ "กว้างและทำงานไวกว่าที่ควร" เช่น ถ้าลูกค้าพิมพ์ว่า "อยากรู้เรื่องผ้า" ขณะที่ดูชุดพยาบาลอยู่ N-gram "อยากรู้" จะไม่ถือว่า match กับชุดพยาบาล แต่ถ้าบังเอิญมีสินค้าชื่อ "เสื้อคอจีน (คนที่**อยากรู้**ไซส์คลิก..)" ในระบบ ระบบจะถือว่า match สินค้าอื่น แล้วสั่งเคลียร์ context ของชุดพยาบาลทิ้งทันที
*   **ผลกระทบ:** ระบบทำลายบริบทง่ายเกินไป พอลอจิกนี้ทำงานพร้อมกับ Prompt ข้อ 2 ทำให้ AI ตอบผิดเพี้ยนไปโดยสิ้นเชิง

### 2. เมื่อมี Context (current_product_id) แล้ว ระบบยังคงวิ่งทำ Keyword Search ซ้ำซ้อน (Keyword Fallback)
*   **สิ่งที่เกิดขึ้น:** ปัจจุบันใน Reseller Bot แม้ `current_product_id` จะมีค่าอยู่แล้ว และรอดจากการถูกเคลียร์ด้วย N-gram มาได้ แต่ถ้าผู้ใช้พิมพ์คำที่ Keyword Search ไป match เจอสินค้าอื่นด้วย ระบบจะดึงสินค้านั้นมารวมใน `products_text` ดึงความสนใจ AI ออกจากสินค้าหลัก
*   **ข้อสังเกตจาก Replit:** ปัญหานี้ได้รับการแก้ไขแล้วในฝั่ง Guest Bot (อ้างอิงบรรทัด ~950) โดยถ้ามี `current_product_id` อยู่แล้ว จะ query เฉพาะสินค้าตัวนั้น ไม่เสียเวลาวิ่ง keyword search ใหม่อีก แต่ในส่วนของ Reseller Bot นี่ยังไม่ได้รับการอัปเดตแก้นี้

### 3. Prompt Rule ห้ามใช้ข้อมูลจากประวัติแชท (ทำงานทวีคูณกับข้อ 1)
(อ้างอิง `app.py` บรรทัดที่ 16517)
ใน System Prompt มีกฎ: `❌ ห้ามใช้ประวัติแชทเดิมเป็นข้อมูลสินค้า ต้องอ้างอิงจากรายการสินค้าด้านล่างเท่านั้น`
*   **เจตนา:** ออกแบบมาดีเพื่อกัน LLM Hallucinate (มโน) สต็อกหรือสเปคสินค้าที่ไม่มีอยู่จริง
*   **ปัญหาที่เกิดร่วมกัน:** เมื่อ Backend เคลียร์ context พลาด (ข้อ 1) ทำให้รายการสินค้าด้านล่างถูกเปลี่ยนไปเป็นตัวอื่น และ Prompt กลับไปห้าม AI ไม่ให้นำประวัติแชทมาแย้ง AI จึงยอมทิ้งบริบทในประวัติทั้งหมดและตอบเพี้ยนตามข้อมูลที่ถูกยัดมาให้ใหม่

## 🛠️ แผนการแก้ไข (Action Plan)

อ้างอิงจากความเห็นของทีม Replit ลำดับความสำคัญในการควรจะเป็นดังนี้:

### ลำดับ 1: แก้ไข N-gram Context Reset (app.py)
*   **ปัญหาที่ต้องแก้:** ไม่ควรรีพึ่ง AI 100% ตามที่ข้อเสนอแนะเดิมเขียนไว้ เพราะ AI อาจจะ "จำสินค้าเก่านานเกินไป" (ไม่ยอมเปลี่ยน state) การมี Backend guard เป็นเรื่องดี แต่ต้องทำให้เข้มงวดกว่านี้ (Strict match)
*   **ทางออก:** ปรับเงื่อนไข N-gram ให้เคลียร์ context **ก็ต่อเมื่อ N-gram นั้นตรงกับชื่อเต็มๆ หรือคำหลักที่มีนัยสำคัญของสินค้าอื่นจริงๆ** ไม่ใช่ส่วนใดส่วนหนึ่งของ description ทั่วไป หรือจำกัดให้ Keyword/N-gram search ใช้เฉพาะเมื่อยังไม่มี `current_product_id`

### ลำดับ 2: ปิด Keyword Search ใน Reseller Bot เมื่อมี Context อยู่แล้ว (เหมือน Guest Bot)
*   **ปัญหาที่ต้องแก้:** ยกเลิกลอจิกที่ดึงสินค้าอื่นๆ มาโชว์แทรกเมื่อการสนทนายังคงมุ่งเป้าที่สินค้าเดิม
*   **ทางออก:** หาก `current_product_id` มีค่า (และยังไม่ถูกเคลียร์อย่างสมเหตุสมผลจากข้อ 1) ให้คิวรีดึงข้อมูลมาแสดงแค่สินค้า ID ตัวนั้นตัวเดียว (`detailed=True`) โดยให้ข้ามส่วนการทำ Keyword Search ค้นหาสินค้าจากคำพูดผู้ใช้ไปเลย

### ลำดับ 3: ปรับแก้ Prompt Rule เรื่องประวัติแชท
*   **ทางออก:** ผ่อนปรนภาษาใน Prompt ไม่ใช่ยกเลิกทั้งหมด เปลี่ยนเป็น:
    > `"✅ ให้ใช้ประวัติแชทเพื่อทำความเข้าใจบริบทว่าลูกค้ากำลังถามถึงสินค้าใด แต่ข้อมูล สต็อก/ราคา/รายละเอียดสินค้า ให้ดึงมาจากรายการสินค้าด้านล่างเท่านั้น ห้ามสร้างข้อมูลขึ้นเอง"`
